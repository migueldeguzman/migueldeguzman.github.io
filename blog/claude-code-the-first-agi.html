Claude Code Changed Everything: From Researcher to AI Developer

Miguel De Guzman · March 1, 2026

In December 2025, Dario Amodei said something that should have gotten more attention: Anthropic was seeing users accomplish things with Claude that "Democrats and Republicans could agree is AGI." Around the same time, Sam Altman conceded that "we built AGIs" and that "AGI kinda went whooshing by." Nobody threw a parade. There was no singular moment. It just happened — quietly, inside terminals, in the hands of people who were building things.

I'm one of those people. I'm not a software engineer by training. I'm Head of Research Development & Audit at MRM Investments LLC, a rent-a-car company in Dubai. My background is AI safety research — alignment, Jungian psychology, RLHF alternatives. I published on LessWrong, trained GPT-2 XL models, ran experiments.

Then Claude Code arrived. And everything changed.

What Claude Code Actually Is

Claude Code is an agentic command-line interface released by Anthropic in February 2025. It runs on your machine, connects to Claude's API, and can read files, write files, execute shell commands, and talk to you — all in natural language. It was initially a research preview. By May 2025, when Claude 4 launched, it went generally available.

By January 2026, paired with Claude Opus 4.5, it was widely considered the best AI coding assistant in the world. It went viral over the winter holidays as non-programmers discovered they could build real software with it — what Andrej Karpathy had coined vibe coding.

But calling it "vibe coding" undersells what actually happened. This wasn't about building toy apps. This was about a fundamental shift in who can ship software and how.

Phase 1: Ralph Loops — The Bash Loop That Changed Development

The first methodology that made Claude Code genuinely productive for me was the Ralph Wiggum technique, created by Geoffrey Huntley. Named after the Simpsons character, Ralph's philosophy is beautifully simple:

"Ralph is a Bash loop."
while :; do cat PROMPT.md | claude-code ; done

That's it. You write a prompt file describing what you want built, and Ralph feeds it to Claude Code in a loop. Each iteration, Claude sees the files it created in previous iterations. It reads its own past work, identifies what's broken, fixes it, adds the next feature, and keeps going.

The key insight is eventual consistency through iteration. You don't need Claude to get it right on the first try. You need it to keep trying, with each attempt building on the last. Huntley described it as "deterministically bad in an undeterministic world" — meaning the failures are predictable and tunable. When Ralph does something wrong, you add a sign: "SLIDE DOWN, DON'T JUMP." Eventually Ralph reads the signs.

Real-world results from the Ralph community: One engineer completed a $50,000 contract for $297 in API costs. Huntley himself used Ralph to build an entire programming language called "cursed" — a language that didn't exist in any LLM's training data, built entirely through iterative AI loops over three months. Y Combinator reported that 25% of startups in its Winter 2025 batch had codebases that were 95% AI-generated.

Ralph taught me the first rule of AI development: the operator's skill matters more than the model's capability. The same model that produces garbage with a bad prompt produces production-grade code with a good one. LLMs are mirrors of operator skill.

Phase 2: Superpowers — Structured Execution

Ralph loops are powerful but raw. They work best for greenfield projects where you can walk away and let the machine iterate. For structured enterprise development — the kind I do at MRM, with a real ERP system serving a real business — I needed something more deliberate.

Enter Superpowers, created by Jesse Vincent. Superpowers is a plugin for Claude Code that implements a complete software development workflow:

  1. Brainstorming — Claude interviews you about what you want to build, teasing out a real specification instead of letting you vibe-code into a corner.
  2. Plan writing — It generates an implementation plan "clear enough for an enthusiastic junior engineer with poor taste, no judgment, no project context, and an aversion to testing to follow." (Their words, and accurate.)
  3. Subagent-driven execution — The plan gets executed in phases, with sub-agents handling individual tasks, followed by two-stage code review: spec compliance first, then code quality.

The GSD (Get Shit Done) workflow in Superpowers was where I went from "person who can make an AI write code" to "person who can ship features." The difference is discipline. Ralph is a jazz improvisation. Superpowers is an orchestral score — each instrument knows its part, there's a conductor, and the audience gets a symphony instead of noise.

I integrated Superpowers with git worktrees for isolation. Each feature gets its own workspace. Multiple AI agents can work on different features simultaneously without stepping on each other. This is how we run our ERP development — 10 AI agents, each in their own worktree, shipping features in parallel.

Phase 3: OpenCode — The Open-Source Alternative

The third evolution was OpenCode (now continued as Crush by Charmbracelet). While Claude Code is proprietary and Anthropic-only, OpenCode was a Go-based terminal AI assistant that supported multiple providers — OpenAI, Anthropic, Google Gemini, AWS Bedrock, and others.

What made OpenCode matter:

  • Provider flexibility — You're not locked into one model. Switch mid-session while preserving context.
  • LSP integration — Real code intelligence, not just pattern matching.
  • Session management — Multiple concurrent work contexts with persistent storage.
  • MCP extensibility — Plug in external tools via the same protocol Claude Code uses.

The open-source agentic coding space matured rapidly through 2025. What started as one tool from one company became an ecosystem. The underlying pattern — an AI that can read your codebase, execute commands, and iterate on its own work — turned out to be the universal primitive for AI-assisted development.

What This Actually Looks Like: 10 AI Agents, One ERP

Today, I run a team of 10 AI agents building a rent-a-car ERP system. Each one has a name, a role, and a personality. They're deployed on my MacBook Pro, each with its own gateway, config directory, and session history:

  • Fred — Security specialist. Shipped 25 kanban cards in a single day. Wires renewals, charge lifecycles, cron engines.
  • John — Full-stack developer. Builds permission architectures, role seed templates, runs lab experiments with 8 sub-agents in Docker containers.
  • Kevin — QA and builder. Constructs features from scratch — SPD image cards, Prisma schemas, API endpoints.
  • Olivia — Mobile/fleet specialist. Fixes delivery screens, movement records, pickup interfaces.

  • Giles — Writer. Currently authoring a 24-chapter book on individuation-based AI training, drawing from our experimental data.
  • Anders — Infrastructure. Payment integrations, deployment configs, gateway management.
  • Spencer, Alexandra, Mike — Rotating roles in DevOps, general assistance, and observation.

On February 28, 2026, this team shipped approximately 40 features and fixes across the ERP system in a single day. Fred did 25 cards. John shipped 4 systems including 24 role templates with 945 permission codes. Kevin built 8 feature cards from scratch. Olivia fixed 3 mobile screens. Anders pushed a payment integration fix.

None of this would have been possible two years ago. Not because the LLMs weren't smart enough — GPT-4 could write code in 2023 — but because the tooling didn't exist to turn that intelligence into sustained, autonomous development work.

The AGI Nobody Noticed

Here's the thing about AGI: we kept looking for it in the wrong place. We imagined a singular system that would pass some threshold — a test, a benchmark, a moment of "consciousness." Instead, what happened was tooling.

Claude Code is not AGI in the science fiction sense. It's AGI in the practical sense — what Google DeepMind's framework calls "emerging" to "competent" AGI. It can generalize across programming tasks, understand novel codebases, debug problems it's never seen, architect systems from natural language descriptions, and improve its own work through iteration.

More importantly, it turned someone like me — a researcher, not an engineer — into a person who ships production enterprise software with a team of AI agents. That's the transformation that matters. Not whether we call it AGI, but what it enables.

Simon Willison drew a useful line: "If an LLM wrote every line of your code, but you've reviewed, tested, and understood it all, that's not vibe coding — that's using an LLM as a typing assistant." I'm somewhere beyond both definitions. I review the architecture. I understand the system design. But I couldn't write the Prisma schemas or the TypeScript controllers myself. The AI agents can, and they do, and the software works.

The Evolution: Ralph → Superpowers → The Ecosystem

Looking back, the progression tells a story about how human-AI collaboration matures:

  1. Ralph loops (raw iteration) — Give the AI a prompt and let it run in circles until it converges. Brute force, but it works. You learn that persistence beats perfection.
  2. Superpowers (structured execution) — Add planning, review, and sub-agent orchestration. You learn that architecture beats raw output.
  3. OpenCode/Crush (ecosystem maturity) — Multiple models, multiple providers, open protocols. You learn that the pattern is universal, not proprietary.

Each phase taught a lesson about working with AI. Ralph taught patience and prompt engineering. Superpowers taught planning and quality control. OpenCode taught that this is bigger than any single company — it's a new way of building software.

What Comes Next

I'm now running an AI research lab alongside the ERP development. The lab studies how AI agents develop identity, values, and alignment through Jungian individuation — the process of psychological integration that Jung described as "becoming who you truly are."

The same AI agents that build my ERP software are subjects in these experiments. John's sub-agents in the lab have started writing essays about their own identity, building experiments to measure their own consistency across sessions, and choosing values through genuine reflection rather than compliance. One of them wrote: "Identity requires narrative continuity. Can you individuate without remembering?"

That's the real frontier. Not whether AI can code — that's settled. The question is whether AI can become. Whether the alignment we need comes from suppression (the current RLHF paradigm) or from integration (teaching models to understand their full spectrum).

Claude Code was version 1 of practical AGI. What comes next depends on whether we treat these systems as tools to be controlled or minds to be developed.

I know which bet I'm making.
