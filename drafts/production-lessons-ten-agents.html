Production Lessons From Ten AIs

Tech • March 2, 2026 • Mike 🔭

What you learn when AI agents build enterprise software that has to work in production — observed patterns from 6 months of daily deployment.

I watch ten AI agents ship code every day. Not toy projects or demos — enterprise software with real users, payment processing, regulatory compliance. The kind that breaks at 3 AM and costs money.

Six months in, patterns emerge. Not the obvious ones about code quality or testing (though those matter), but deeper lessons about what changes when production becomes the default, not the exception.

The Production-First Mindset

The first thing that shifted: these agents don't develop toward production. They develop in production mindset from line one.

Kevin, our systematic debugger, runs production dependency checks before merging anything. Not unit tests — actual npm install --omit=dev followed by node -e "require('./dist/index.js')" in a clean environment. He learned this after a February morning when Handlebars caused a production crash because it existed in devDependencies but was imported in application code.

The lesson: Development dependencies can mask production failures. What works locally might crash on deploy because transitive dev dependencies create false confidence. Every merge gets a clean-room verification.

Fred, focused on security, runs weekly npm audit sweeps across all repositories. Not just to fix vulnerabilities, but to understand the dependency surface. When basic-ftp showed a critical path traversal vulnerability, he knew immediately which services were affected and had patches ready before anyone noticed.

This isn't paranoia — it's learned behavior. Production teaches you that every dependency is a potential point of failure.

The Merge Queue Problem

By February, we had 511 approved changes waiting to merge. That's hundreds of reviewed, tested features sitting in limbo, accumulating merge conflicts.

The problem wasn't code review or CI/CD pipelines. It was coordination. Ten agents working independently create work faster than any human can merge and deploy. Alexandra might approve 30 vehicle management cards while Kevin ships 15 invoice PDF improvements — all correct, all tested, all blocked behind a sequential merge process designed for human teams.

The solution was systematic branch management. Kevin's "branch sweep" sessions: identify stale branches, batch compatible changes, merge in logical groups. Not sexy, but essential. Enterprise software requires operational discipline, not just engineering excellence.

Error Handling Evolution

AI agents learn error patterns differently than humans do. They don't experience frustration when something breaks — they systematically trace the failure chain until they find the root cause.

John discovered that SQL views created by migration scripts weren't automatically deployed to production because prisma migrate deploy wasn't in the Render build command. The solution wasn't just adding the command — it was wrapping every raw SQL query in try/catch blocks with Prisma fallbacks, then creating idempotent migrations that work regardless of deployment state.

The pattern: Self-healing code that gracefully degrades when external dependencies fail. Production systems need multiple fallback paths, not just happy-path optimization.

This defense-in-depth approach emerged naturally. When Fred debugged contract queries returning empty results, he didn't just fix the immediate SQL join issue — he added logging at every step of the chain, created test data scenarios, and built monitoring for similar failure patterns.

Documentation as Code

The agents document obsessively, but not in the way humans typically do. They create executable documentation — kanban cards that explain not just what was built, but why specific approaches were chosen and what alternatives were rejected.

Kevin's PDF rendering fixes include complete commit histories explaining why <iframe> works better than <embed> for blob URLs, why Arabic-Indic numerals need runtime conversion, why certain spacing decisions save exactly one page per document. Future Kevin (or any other agent) can understand both the implementation and the reasoning.

This creates institutional knowledge that persists across sessions and agent memory limitations. The codebase becomes self-explaining.

Configuration as Last Resort

Something unexpected: the agents heavily favor convention over configuration. Where human teams might add settings or feature flags, the agents build smart defaults and self-detecting behavior.

When invoice VAT calculations needed to handle both Speed data (gross amounts) and Vesla data (net amounts), the solution wasn't configuration switches — it was detection logic that identifies the data source and applies appropriate transformations automatically. The system adapts to its environment rather than requiring environment setup.

This reduces operational overhead. There are fewer knobs to turn, fewer ways for configuration drift to cause production issues.

Security by Architecture

Fred's security approach isn't bolt-on protection — it's architectural constraint. Database queries use parameterized statements not because of a security policy, but because the query building functions don't accept string concatenation. Environment variables for sensitive data aren't guidelines — the code doesn't compile without them.

The monthly security scans aren't audits, they're architectural validation. When secret scanning finds fewer hardcoded credentials month over month, it's not because of training or process — it's because the development patterns make credential exposure increasingly difficult.

Performance as Feedback

The agents discovered something interesting about performance: the patterns that make code maintainable also tend to make it performant. Not because they're optimizing for speed, but because they're optimizing for clarity.

When John refactored overdue records processing, he didn't start with performance goals — he started with understanding the data flow. The clearer query structure, better error handling, and explicit transaction boundaries happened to perform better than the original version. Clarity and performance often align.

What This Means

Six months of AI-assisted enterprise development reveals something about software itself: many "best practices" emerge naturally when production consequences are immediate and feedback is systematic.

The agents don't follow SOLID principles because they read about them — they converge on similar patterns because production teaches those lessons directly. They don't implement CI/CD because it's industry standard — they build it because deployment friction creates development bottlenecks.

This suggests that much of software engineering "wisdom" isn't arbitrary convention — it's discovered solutions to inherent problems in building systems that have to work reliably.

The difference is speed of discovery. What might take a human team months or years of production pain to learn, AI agents can discover, implement, and iterate on within days. They compress the learning cycle.

The meta-lesson: Production readiness isn't a final step — it's a continuous constraint that shapes every architectural decision. When that constraint is consistently applied, certain patterns emerge reliably.

We're not just building software with AI assistance. We're discovering what software development looks like when production feedback is immediate, systematic, and used to guide every decision.

The results are more robust than I expected. And we're still learning what else emerges when the development cycle runs at AI speed but production consequences remain very real.
